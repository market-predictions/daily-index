# FX Runtime Split — Split-Test Entrypoint

This file is the **entrypoint** for running the FX prompt in the new split architecture.

## Purpose
Run the existing FX methodology, research logic, delivery expectations, and executive look & feel **as-is**, but sourced from the split files instead of directly from `fx.txt`.

## Project-context note
This file is a **task-specific split-test runtime entrypoint**.
It is **not** the default standing ChatGPT Project context.
Read this file only **after**:
1. `control/SYSTEM_INDEX.md`
2. `control/CURRENT_STATE.md`
3. `control/NEXT_ACTIONS.md`

Use the control layer first for operating context, and use the live GitHub versions of the split files as the active source during split evaluation.

## Non-change rule
Do not change:
- research logic
- decision methodology
- scoring thresholds
- state-authority logic
- technical-overlay role
- report structure
- delivery requirements
- executive presentation standard

This is an **as-is split test**, not a content revision.

## Authoritative read order
Read and combine these files in this exact order before executing the run:
1. `prompts/as_is_split/04_OPERATIONAL_RUNBOOK.md`
2. `prompts/as_is_split/02_INPUT_STATE_CONTRACT.md`
3. `prompts/as_is_split/01_DECISION_FRAMEWORK.md`

## Cross-check rule
Use `prompts/as_is_split/05_SECTION_MAP.md` only as an architectural reference.

## Output path for split tests
When you run this split-test setup, write the resulting FX report to:
- `output_split_test/weekly_fx_review_YYMMDD.md`
- or, if needed on the same day, `output_split_test/weekly_fx_review_YYMMDD_NN.md`

Use the same naming/version discipline as the production report family.

## Production safety rule
- Keep `fx.txt` untouched as the current production source.
- Use this split runtime only for comparison runs.
- Do not overwrite production prompt files during split testing.

## Delivery note
A separate split-test workflow may render and send the latest report from `output_split_test/` for output comparison.
That workflow is for evaluation only and must not replace the production workflow.
